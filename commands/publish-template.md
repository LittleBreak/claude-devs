用户输入的信息为：`$ARGUMENTS`

## 数据源

在生成模版之前，先读取数据源文件获取补充信息：

- 读取 `data/publish-template-data.md` 文件内容，作为辅助数据源

数据源中可能包含人员信息、项目信息等。当用户输入的信息中出现的关键词能在数据源中**精确匹配**到对应内容时，可以用数据源中的信息来补充模版字段。

## 严格规则

1. **只允许根据用户输入的信息和数据源中的信息进行填写**，绝对不允许通过推测来补充信息
2. 如果某个字段没有匹配的信息，保留该字段为 `-`（横杠）
3. 只输出模版内容，不要输出任何额外的说明、解释或建议
4. 日期格式为 `xx.xx日`，如果用户提供了日期则填入，否则留空为 `xx.xx日`

## 输入解析

用户输入可能包含多个需求的信息。需要先识别并解析出所有需求，然后合并到一个模版中输出。

### 项目名称识别

用户输入的第一行通常是项目的英文名称（如 `wmmos`）。识别后将其作为每个需求描述的后缀补充，格式为 `需求描述（项目名）`。

### 多需求识别

用户输入中通常以空行分隔不同需求的信息块。每个需求信息块可能包含：需求描述、研发人员、测试人员、合并请求链接等。请根据上下文语义识别各信息块的边界。

### 合并规则

当输入包含多个需求时，将所有需求合并到**一个**模版中输出：

- **需求**：多个需求用序号列出，每个需求单独一行，格式为 `序号. 需求描述`
- **研发**：合并所有研发人员，去重后用 `、` 分隔
- **测试**：合并所有测试人员，去重后用 `、` 分隔
- **产品**：合并所有产品人员，去重后用 `、` 分隔
- **合并请求**：所有 MR/PR 链接逐行列出
- **其他字段**：正常填写，如有冲突以最后出现的为准

## 模版

```
【xx.xx日上线报备】
需求：xxxxxxxxxxxx
技术方案：-
研发: -
测试：-
产品：-
合并请求：-
发布方式：行云2.0
上线方案：-
盯盘：-
```

## 字段映射规则

| 模版字段   | 说明                                           |
| ---------- | ---------------------------------------------- |
| 日期       | 上线日期，格式为 `xx.xx日`                      |
| 需求       | 需求名称或描述，多个需求时用序号逐行列出          |
| 技术方案   | 技术方案链接或描述                               |
| 研发       | 研发人员，多人用 `、` 分隔                       |
| 测试       | 测试人员，多人用 `、` 分隔                       |
| 产品       | 产品人员                                        |
| 合并请求   | MR/PR 链接，多个时逐行列出                       |
| 发布方式   | 默认为 `行云2.0`，用户指定则替换                  |
| 上线方案   | 上线方案描述或链接                               |
| 盯盘       | 盯盘监控链接，从数据源中匹配                      |

## 输出要求

- 直接输出纯文本模版内容，不要用代码块包裹
- 不要输出任何额外文字
