# 验证问题记录

你的职责是：在开发任务验证阶段发现并修复问题后，将问题信息结构化地记录到对应的 task 文档中。

## 触发条件

当满足以下任一条件时，执行此 skill：

1. **自动触发**：在当前对话中，你刚刚修复了一个验证阶段发现的问题（无论是自动发现还是用户报告），且问题已确认修复
2. **手动触发**：用户通过 `/verify-log` 主动调用，或说"记录一下这个问题"、"更新问题记录"等

## 执行流程

### Step 1: 定位 task 文档

- 如果用户提供了 `$ARGUMENTS`，直接使用该路径
- 否则，在当前项目中查找 `*-task.md` 文件，优先选择最近修改的
- 如果找到多个，询问用户使用哪个

### Step 2: 收集问题信息

从当前对话上下文中提取以下信息（如果信息不完整，向用户确认）：

- **问题描述**：出了什么问题，现象是什么
- **关联任务**：对应 task 文档中的哪个任务编号/名称
- **问题原因**：根本原因分析
- **解决方案**：具体做了什么来修复
- **改动范围**：影响了哪些模块/功能，对项目稳定性或安全性的影响评估
- **发现方式**：自动检测 / 手动发现

### Step 3: 更新文档

读取 task 文档，按以下规则更新：

1. **如果文档中已有「问题记录」章节**：在该章节末尾追加新的问题条目，编号递增
2. **如果文档中没有「问题记录」章节**：在文档末尾添加分隔线和「问题记录」章节

使用 [reference.md](reference.md) 中定义的模板格式记录。

### Step 4: 确认

向用户展示已记录的问题摘要，确认更新完成。

## 注意事项

- 语言：使用中文记录
- 只记录已确认修复的问题，未修复的不记录
- 问题原因要写根本原因，不要只写表象
- 改动范围要从模块/功能层面描述，而非逐一列出文件
- 影响评估要客观，区分"无影响"、"低影响"、"中等影响"、"高影响"
